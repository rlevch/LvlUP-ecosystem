# Требования к новой ролевой модели Академии (v2)

> **Дата:** 2026-03-29
> **Статус:** Черновик
> **Автор:** Roman + Claude Architect

---

## 1. Ключевое изменение: централизация профилей

**Было:** Профили пользователей частично хранились в контексте онлайн-школ (tenant-scoped).

**Стало:** Все профили пользователей с любыми ролями хранятся централизованно на уровне Академии (public/platform schema). Каждый пользователь имеет единый личный кабинет, наполнение которого динамически зависит от его активных ролей.

### Принципы

- Один пользователь = один `public.profiles` + один `auth.users` (Supabase GoTrue).
- Роли — это **связи**, а не свойства пользователя. Пользователь может иметь несколько ролей одновременно.
- Личный кабинет — единая точка входа с переключением контекста между ролями.

---

## 2. Ролевая модель

### 2.1 Глобальные роли (уровень Академии)

| Роль | Код | Кто назначает | Описание |
|------|-----|---------------|----------|
| Администратор сервиса | `service_admin` | Вручную / seed | Полные права ко всему сервису Академии, всем школам и оплатам. Верификация владельцев школ. |
| Студент | `student` | Самостоятельная регистрация | Оплачивает и обучается на курсах из любых школ. |

### 2.2 Роли с привязкой к школе (tenant-scoped)

| Роль | Код | Кто назначает | Описание |
|------|-----|---------------|----------|
| Владелец онлайн-школы | `school_owner` | `service_admin` | Создание и управление своими школами. Юрлицо/самозанятый. Работает после верификации. |
| Преподаватель | `school_instructor` | `school_owner` (через invite-link) | Ведёт супервизии онлайн, проверяет ДЗ, управляет расписанием своих курсов. |
| Куратор | `school_curator` | `school_owner` (через invite-link) | Рассылает сообщения студентам, управляет сертификатами, контролирует оплаты, формирует ссылки на курсы. |

### 2.3 Совмещение ролей

Один пользователь может совмещать любые роли. Примеры допустимых комбинаций:

- `service_admin` + `school_owner` — администратор, который также владеет школой
- `school_owner` (школа A) + `school_instructor` (школа B) — владелец одной школы и преподаватель в другой
- `student` + `school_curator` (школа X) — студент в одних школах и куратор в другой
- `school_instructor` (школа A) + `school_instructor` (школа B) — преподаватель в нескольких школах

**Ограничения назначения:**
- Роли `service_admin` и `school_owner` может назначить **только** `service_admin`.
- Роли `school_instructor` и `school_curator` назначает `school_owner` через ссылку-приглашение.
- Роль `student` присваивается автоматически при регистрации.

## 3. Детальное описание ролей

### 3.1 Администратор сервиса (`service_admin`)

**Права:**
- Полный доступ ко всем школам, курсам, пользователям
- Управление всеми оплатами и финансовой отчётностью
- Верификация владельцев школ (проверка документов, подписание агентского договора)
- Назначение ролей `service_admin` и `school_owner`
- Блокировка/разблокировка пользователей и школ
- Просмотр аналитики по всей платформе

**Личный кабинет:**
- Дашборд с метриками платформы (школы, пользователи, доход)
- Очередь верификации владельцев школ
- Список всех школ с возможностью перехода в любую
- Управление пользователями и ролями
- Финансовые отчёты и управление оплатами

### 3.2 Владелец онлайн-школы (`school_owner`)

**Предусловия для работы:**
1. Юридическое лицо или самозанятый (ИНН, ОГРН/ОГРНИП обязательны)
2. Загрузка документов: учредительные документы / справка о самозанятости
3. Подписание агентского договора с платформой
4. Верификация администратором сервиса (ручная проверка)

**Статусы верификации:** `pending` → `documents_submitted` → `under_review` → `verified` / `rejected`

**Права (после верификации):**
- Создание и управление своими онлайн-школами
- Управление списком преподавателей и кураторов
- Формирование invite-ссылок для преподавателей и кураторов (с указанием роли)
- Формирование учебных групп студентов
- Управление расписанием групп
- Управление оплатой: тарифы курсов, промокоды, скидки
- Просмотр финансовой аналитики своих школ

**Личный кабинет:**
- Список своих школ с быстрым переключением
- Для каждой школы: курсы, группы, команда, финансы
- Панель верификации (статус, загрузка документов)
- Настройки школы (брендинг, домен, описание)

### 3.3 Преподаватель онлайн-школы (`school_instructor`)

**Как попадает:** Переходит по invite-ссылке от владельца школы → авторизуется → автоматически получает роль `school_instructor` в этой школе.

**Права:**
- Ведение супервизий в онлайн-формате (LiveKit видео-сессии)
- Проверка домашних заданий по курсам, в которых преподаёт
- Изменение расписания курса, в котором преподаёт
- Просмотр списка студентов своих курсов
- Общение со студентами в чате курса

**Личный кабинет:**
- Список школ, в которых преподаёт
- Расписание занятий (календарь)
- Список ДЗ на проверку
- Предстоящие супервизии
- Список студентов по курсам

### 3.4 Куратор онлайн-школы (`school_curator`)

**Как попадает:** Переходит по invite-ссылке от владельца школы → авторизуется → автоматически получает роль `school_curator` в этой школе.

**Права:**
- Рассылка сообщений студентам разных курсов
- Управление созданием и выдачей сертификатов
- Контроль оплаты курсов студентами
- Формирование ссылки на курс для отправки студентам
- Просмотр информации о студентах и их прогрессе

**Механизм ссылки на курс:**
1. Куратор в ЛК выбирает курс → генерирует ссылку (формат: `https://levelup-academy.ru/invite/course/{token}`)
2. Копирует ссылку → отправляет студенту (email, мессенджер, и т.п.)
3. Студент переходит по ссылке → авторизуется (или регистрируется)
4. Курс автоматически добавляется в ЛК студента:
   - **Бесплатный курс** → доступен сразу (с учётом расписания)
   - **Платный курс** → отображается с кнопкой оплаты, доступен после оплаты (с учётом расписания)

**Личный кабинет:**
- Список школ, в которых куратор
- Студенты по курсам (с фильтрами по оплате, прогрессу)
- Управление сертификатами
- Рассылки и уведомления
- Контроль оплат

### 3.5 Студент (`student`)

**Как попадает:** Самостоятельная регистрация на платформе или переход по ссылке на курс.

**Права:**
- Просмотр каталога курсов из любых школ
- Оплата курсов
- Обучение на курсах (просмотр уроков, выполнение ДЗ, прохождение тестов)
- Получение сертификатов
- Участие в видео-сессиях (супервизии)

**Личный кабинет:**
- Мои курсы (активные, завершённые, ожидающие оплаты)
- Расписание занятий
- Онлайн-школы, в которых обучается
- История оплат и квитанции
- Полученные сертификаты
- Прогресс по курсам

## 4. Ключевые механизмы

### 4.1 Invite-ссылки

**Для команды школы (преподаватели, кураторы):**
```
https://levelup-academy.ru/invite/team/{token}
```
- Генерирует владелец школы
- Содержит: `school_id`, `role` (instructor/curator), `expires_at`, `max_uses`
- При переходе: авторизация → автоматическое добавление в `school_members` с нужной ролью

**Для студентов (ссылка на курс):**
```
https://levelup-academy.ru/invite/course/{token}
```
- Генерирует куратор
- Содержит: `course_id`, `school_id`, `expires_at`, `max_uses`, `promo_code` (опционально)
- При переходе: авторизация → добавление курса в ЛК → оплата (если платный) → доступ по расписанию

### 4.2 Верификация владельца школы

**Процесс:**
1. Пользователь подаёт заявку на роль `school_owner`
2. Загружает документы (ИНН, ОГРН, учредительные документы)
3. Подписывает агентский договор (электронная подпись или скан)
4. `service_admin` проверяет документы
5. При успешной проверке: статус → `verified`, пользователь может создавать школы
6. При отклонении: статус → `rejected` с комментарием, возможность повторной подачи

### 4.3 Переключение ролевого контекста в ЛК

Единый ЛК с боковым меню, где отображаются доступные контексты:

```
┌─────────────────────────────────┐
│  👤 Иван Петров                 │
│  ─────────────────────          │
│  📚 Мои курсы (студент)        │
│  🏫 Школа "Коучинг PRO"        │
│     └─ Владелец                 │
│  🏫 Школа "МастерКласс"        │
│     └─ Преподаватель            │
│  ⚙️ Администрирование          │
│     └─ Админ сервиса            │
└─────────────────────────────────┘
```

## 5. Влияние на существующую архитектуру

### 5.1 Что меняется в БД

| Область | Текущее состояние | Что нужно |
|---------|-------------------|-----------|
| `users.role[]` | Глобальные роли в массиве | Заменить на таблицу `user_roles` |
| `tenant.school_team_members` | Роли привязаны к школе | Переименовать, расширить логику (invite-ссылки, статус верификации) |
| Профили | `public.profiles` уже централизованы | Добавить поля для юрлица (ИНН, ОГРН, тип) |
| Верификация | `platform.verification_documents` | Расширить для владельцев школ + агентский договор |
| Invite-ссылки | Нет | Новая таблица `invite_links` |
| Привязка курсов к студентам | `academy.enrollments` | Добавить `source` (invite_link, catalog, manual) |

### 5.2 Что меняется в API

- Новые эндпоинты для invite-ссылок (генерация, валидация, использование)
- Рефакторинг RBAC middleware: проверка ролей через `user_roles` + `school_members`
- Эндпоинты верификации владельца школы
- Эндпоинт переключения контекста (текущая школа / роль)

### 5.3 Что меняется во фронтенде

- Единый ЛК с переключением контекста ролей
- Динамическое меню на основе ролей пользователя
- Страницы верификации для владельца школы
- UI для генерации и управления invite-ссылками
- Логика обработки invite-ссылок при авторизации

## 6. Диаграмма новой ролевой модели

```
                    ┌──────────────────┐
                    │   auth.users     │
                    │   (Supabase)     │
                    └────────┬─────────┘
                             │ 1:1
                    ┌────────▼─────────┐
                    │ public.profiles   │
                    │ (централизованно) │
                    │ + legal_entity    │
                    └────────┬─────────┘
                             │ 1:N
              ┌──────────────▼──────────────┐
              │       user_roles             │
              │  (global: service_admin,     │
              │         student)             │
              └──────────────┬──────────────┘
                             │
              ┌──────────────▼──────────────┐
              │      school_members          │
              │  (tenant-scoped:             │
              │   school_owner,              │
              │   school_instructor,         │
              │   school_curator)            │
              │  + status (для owner:        │
              │    verified/pending/rejected) │
              └──────────────┬──────────────┘
                             │ N:M
              ┌──────────────▼──────────────┐
              │      tenant.schools          │
              └─────────────────────────────┘
```
